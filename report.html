Root Cause — Why These Issues Occur

NULL user_pseudo_id: Safari ITP blocks cookies on first visit. GA4 falls back to session-scoped IDs, but misconfigured GTM can drop the param entirely.

Duplicate transactions: Network retry on payment confirmation page fires the purchase event twice before GTM deduplication check loads.

String revenue: GTM Variable type set to "DOM Element" (returns text) instead of "JavaScript Variable" (returns number). Classic tag misconfiguration.

dataLayer Push — Purchase Event (Full Schema)

// Fires on order confirmation page load — before GTM fires purchase tag window.dataLayer = window.dataLayer || []; window.dataLayer.push({ 'event': 'purchase', 'ecommerce': { 'transaction_id': 'T-38291', // unique per order — dedup key 'value': 127.75, // FLOAT — not "$127.75" 'currency': 'USD', 'coupon': 'BLACKFRI20', 'items': [{ 'item_id': 'GGOEGAAX0104', 'item_name': 'Google Unisex Eco Tee', 'item_category': 'Apparel', 'price': 25.55, 'quantity': 5, 'coupon': 'BLACKFRI20' }] }, 'user_id': 'U-743801', // first-party, not hashed at push time 'consent': { 'analytics': 'granted', 'ads': 'granted' } });
Why value must be a number, not a string

GTM variables of type "DOM Element" return the visible text content of an HTML node — e.g. "$127.75". GA4 purchase tags require value to be a JavaScript number. If a GTM variable reads from the DOM, the dollar sign and comma will make GA4 silently drop the parameter, recording value: 0 in BigQuery. The correct approach: use a "JavaScript Variable" reading from the dataLayer object — never a DOM scrape for monetary values.
