Chapter One
Why GA4 Matters
GA4 Ecommerce Intelligence Hub · Anthony Apollis
Universal Analytics Sunset → GA4 Event Model

The Biggest Analytics
Platform Shift in a Decade

On 1 July 2023, Google permanently shut down Universal Analytics data processing. Every digital analytics team in the world was forced to migrate or go dark. This ebook documents a real GA4 implementation that ran in parallel with UA for two years, capturing the full migration across 13,842,960 events and $891,480 in revenue.

Universal Analytics (Sunset)

UA was built on a session-based, hit-centric model. Every interaction was a "hit" — pageview, event, transaction — counted inside a session. Sessions reset at midnight, at source changes, and after 30 minutes of inactivity. This created attribution headaches and made cross-device tracking nearly impossible without User ID override.

UA's BigQuery export was limited to 360 (paid) customers only — standard at $150K+/year. Standard UA had no raw data access: only sampled reports. Custom dimensions were limited to 20 hit-scoped and 5 session/user-scoped.

UA Critical Limitations

Sessions reset on source change · No cross-device native · BigQuery export requires 360 · Sampling above 500K sessions · Data permanently deleted July 2024 · No event-parameter nesting · 20 custom dimensions max

GA4 — The New Model

GA4 is built on an event-based, user-centric model. Every interaction is an event. Sessions are derived from events, not the reverse. The schema is flat — every event carries its own parameters as key-value pairs in an event_params[] ARRAY column in BigQuery.

Crucially, GA4 offers free BigQuery export to all properties. This is the single biggest capability unlock: direct SQL access to every raw event without sampling, without paying for 360.

GA4 Capabilities Unlocked

Free BQ export to all users · 25 custom dimensions · 50 custom metrics · Cross-device via User ID · Consent Mode V2 native · ML-powered predictive audiences · App + Web unified property · event_params[] nesting

The free BigQuery export transforms GA4 from a reporting tool into a full data platform — enabling ML training, LTV modelling, custom attribution, and real-time pipeline integrations that were previously exclusive to 360 customers paying $150K+/year. dataLayer.push() · GTM Variables · GOOGTAG vs GAAWE

How the Browser Talks
to Google Analytics

The dataLayer is the contract between front-end developers and the analytics implementation. Every ecommerce event must be pushed to the dataLayer before GTM can capture and forward it to GA4. Understanding this architecture is the most important skill for an analytics engineer.

dataLayer.push() — Full Purchase Event Schema

// STEP 1: Clear previous ecommerce data (REQUIRED before every push) window.dataLayer = window.dataLayer || []; dataLayer.push({ ecommerce: null }); // STEP 2: Push purchase event with complete GA4 ecommerce schema dataLayer.push({ event: 'purchase', // matches GTM Custom Event trigger name ecommerce: { transaction_id: 'T-38291', // DLV: ecommerce.transaction_id — unique order ID value: 25.99, // DLV: ecommerce.value — FLOAT64 not string currency: 'USD', // DLV: ecommerce.currency — ISO 4217 coupon: 'SAVE10', // optional discount code shipping: 5.99, tax: 2.34, items: [{ item_id: 'GGOEGAAX0104', item_name: 'Google Unisex Eco Tee', item_category: 'Apparel', item_brand: 'Google', price: 25.99, quantity: 1, index: 0 }] } });

Variable Hierarchy in GTM

📦
DataLayer Variables (DLV)
Read from window.dataLayer using dot-notation. DLV "ecommerce.value" reads dataLayer[n].ecommerce.value. Used for all ecommerce data: transaction_id, items[], value, currency.
🔗
URL Variables (Built-in)
Page URL, Page Path, Page Hostname, Query. GTM reads these automatically from the browser location object. No dataLayer push needed.
⚡
JavaScript Variables
Custom JS that returns a value at trigger time. Used for: cookie consent status (window.cookieConsent), login state, A/B test bucket. Runs in GTM sandbox.
🍪
1st-Party Cookie Variables
Reads named cookies: GA client_id (_ga), consent prefs. Critical for Safari ITP — server-side GTM required for accurate cookie lifespan.

GOOGTAG vs GAAWE Tags

GOOGTAG — GA4 Configuration Tag

Fires once per page on All Pages trigger. Loads the gtag.js library, sets Measurement ID (G-XXXXXXXXXX), initialises Consent Mode, sets cookie_flags. Must fire BEFORE all GAAWE tags via GTM tag sequencing (setup tag).

GAAWE — GA4 Event Tag

Fires per-event on Custom Event triggers. Sends named GA4 events (purchase, add_to_cart, view_item) with parameters mapped from DLV variables. One GAAWE per event type. The "Event Name" field maps to GA4's event_name column in BigQuery.

Critical: Clear ecommerce before each push

Always push { ecommerce: null } before each ecommerce push. Without this, GTM merges consecutive pushes — a view_item transaction_id can bleed into a purchase event, causing duplicated or wrong data in BigQuery.

GA4's free BigQuery export writes one partitioned table per day: analytics_XXXXXXXXXX.events_YYYYMMDD. Each row is one event. Repeated ARRAY fields (event_params, user_properties, items) require the UNNEST pattern to access. This is the most critical SQL skill for any GA4 analytics engineer.

BATCH vs INTERACTIVE Queries

BATCH (QueryPriority.BATCH)

Queued in shared slot pool. Starts within 24h (typically under 6h). Same byte billing as INTERACTIVE on on-demand pricing. Free on flat-rate slots (consumes same bytes, no surcharge). Best for: nightly ML pipelines, aggregation jobs, bulk BQ exports.

INTERACTIVE (default)

Starts immediately. On-demand: $6.25/TB scanned (first 1TB/month free). Best for: DebugView queries, ad-hoc exploration, real-time dashboard queries. Use for anything with an SLA under 1 hour.

Essential SQL Patterns — UNNEST, Wildcard Tables, Session Aggregation

-- Pattern 1: Extract event_params value (most-used GA4 BigQuery pattern) SELECT event_date, user_pseudo_id, (SELECT value.string_value FROM UNNEST(event_params) WHERE key = 'transaction_id') AS transaction_id, (SELECT value.double_value FROM UNNEST(event_params) WHERE key = 'value') AS revenue FROM `project.analytics_XXX.events_*` -- wildcard table = all date shards WHERE _TABLE_SUFFIX BETWEEN '20230101' AND '20231031' -- ALWAYS prune partitions AND event_name = 'purchase'; -- Pattern 2: UNNEST items array (ecommerce line items) SELECT event_date, (SELECT value.string_value FROM UNNEST(event_params) WHERE key='transaction_id') AS txn, item.item_id, item.item_name, item.quantity, item.price FROM `project.analytics_XXX.events_*`, UNNEST(items) AS item WHERE _TABLE_SUFFIX = '20231031' AND event_name = 'purchase'; -- Pattern 3: Session aggregation (count events + purchases per session) SELECT user_pseudo_id, (SELECT value.int_value FROM UNNEST(event_params) WHERE key='ga_session_id') AS session_id, COUNT(*) AS events_in_session, COUNTIF(event_name='purchase') AS purchases FROM `project.analytics_XXX.events_*` WHERE _TABLE_SUFFIX BETWEEN '20230101' AND '20231031' GROUP BY 1, 2;
Partition Pruning — Cost Control Rule #1

Always filter on _TABLE_SUFFIX in wildcard queries. Without it, BigQuery scans every daily shard since GA4 began exporting — potentially terabytes. Use event_date for display filtering but _TABLE_SUFFIX for cost control. These are different: _TABLE_SUFFIX prunes at the storage layer; event_date filters after scanning.

dbt Staging Model — Core SQL

-- models/staging/stg_ga4_events.sql WITH raw AS ( SELECT CASE WHEN REGEXP_CONTAINS(event_date, r'^\d{4}-\d{2}-\d{2}$') THEN PARSE_DATE('%Y-%m-%d', event_date) WHEN REGEXP_CONTAINS(event_date, r'^\d{2}/\d{2}/\d{4}$') THEN PARSE_DATE('%d/%m/%Y', event_date) ELSE DATE(TIMESTAMP_MICROS(event_timestamp)) END AS clean_date, LOWER(TRIM(event_name)) AS event_name, SAFE_CAST( REGEXP_REPLACE(revenue_raw, r'[^0-9.]', '') AS FLOAT64) AS revenue, user_pseudo_id, event_timestamp, ROW_NUMBER() OVER ( PARTITION BY transaction_id ORDER BY event_timestamp DESC ) AS row_num FROM {{ source('ga4', 'events_*') }} ) SELECT * EXCEPT(row_num) FROM raw WHERE row_num = 1 -- deduplicate AND event_name IS NOT NULL

Consent Mode V2 Implementation (GTM Custom HTML Setup Tag)

// Setup tag fires BEFORE GOOGTAG via GTM tag sequencing window.dataLayer = window.dataLayer || []; function gtag() { dataLayer.push(arguments); } // V2 requires all 4 ad_* parameters (ad_storage, ad_user_data, ad_personalization are new in V2) gtag('consent', 'default', { 'analytics_storage': 'denied', // blocks GA4 cookies until consent granted 'ad_storage': 'denied', // blocks Google Ads cookies 'ad_user_data': 'denied', // NEW in V2 — user data sent to Google for ads 'ad_personalization': 'denied', // NEW in V2 — personalised ads targeting 'wait_for_update': 500 // 500ms wait for CMP to update consent }); // After user accepts (CE trigger: consent_updated fires consent.update) gtag('consent', 'update', { 'analytics_storage': 'granted', 'ad_storage': 'granted', 'ad_user_data': 'granted', 'ad_personalization': 'granted' });
Tag Sequencing — Critical

Set the Consent Mode V2 Custom HTML tag as a "Setup Tag" for GOOGTAG. Set GOOGTAG as "Setup Tag" for all GAAWE event tags. This guarantees: Consent init → GA4 config → Event tags. Without this order, purchase events may fire before the G-ID is configured, causing silent data loss.

Safari ITP — Server-Side GTM Required

Safari Intelligent Tracking Prevention limits 1st-party cookies set via JavaScript to 7 days (and 24h for cookies set from script includes). 18.4% of Safari sessions in this dataset show session_number=1 on return visits — clear evidence of cookie deletion. Server-side GTM extends cookie lifetime to 400 days by setting cookies at the server level.

BigQuery Schema Evolution

Safe Changes (backward-compatible)

Adding new event_params keys: safe — new keys appear in new shards, old shards return NULL. Adding new GTM tags: safe if existing params unchanged. Use SAFE_CAST + COALESCE in all dbt models to handle NULL columns in historical partitions.

Breaking Changes (avoid)

Renaming event params: breaks all queries using old name. Changing param data type (STRING to FLOAT): SAFE_CAST returns NULL on old data. Never ALTER production BQ tables — use dbt migrations with backward-compatible column additions only.
