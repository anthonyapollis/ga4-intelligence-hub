Consent Mode v2 — Implementation

Consent Mode v2 must fire before the GA4 Config tag. The GTM container loads consent defaults in the gtm.js script using a custom HTML tag with firing priority 999 (highest).

// Fires BEFORE GA4 Config tag (priority 999)
window.dataLayer = window.dataLayer || [];
function gtag(){dataLayer.push(arguments);}

gtag('consent', 'default', {
  analytics_storage:      'denied',
  ad_storage:             'denied',
  ad_user_data:           'denied',
  ad_personalization:     'denied',
  wait_for_update:        500  // ms to wait for CMP
});

DataLayer Pattern — Ecommerce Events

// Always clear before pushing — prevents data bleeding
window.dataLayer.push({ ecommerce: null });

window.dataLayer.push({
  event: 'purchase',
  ecommerce: {
    transaction_id: 'GMS-20231015-001',
    currency:       'USD',
    value:          149.96,
    tax:            15.00,
    shipping:       0.00,
    items: [{
      item_id:       'SKU-001',
      item_name:     'Google Hoodie',
      item_category: 'Apparel',
      quantity:      2,
      price:         74.98
    }]
  }
});
Why clear ecommerce: null first? Without the null push, data from a previous event can bleed into the next push. A view_item → add_to_cart sequence without clearing would have the wrong items array on add_to_cart. This is the most common GTM ecommerce bug.

BigQuery Funnel Query

WITH funnel AS (
  SELECT
    COUNTIF(event_name = 'session_start')     AS sessions,
    COUNTIF(event_name = 'view_item')         AS view_item,
    COUNTIF(event_name = 'add_to_cart')       AS add_to_cart,
    COUNTIF(event_name = 'begin_checkout')    AS begin_checkout,
    COUNTIF(event_name = 'purchase')          AS purchase
  FROM `analytics_XXXXXXXXX.events_*`
  WHERE _TABLE_SUFFIX BETWEEN '20201101' AND '20231031'
)
SELECT *, ROUND(purchase * 100.0 / sessions, 2) AS overall_cr
FROM funnel;

GA4 BigQuery Export Schema

GA4 exports one table per day to BigQuery in the format analytics_PROPERTYID.events_YYYYMMDD. The key complexity is nested STRUCT fields — especially event_params which requires UNNEST to query.

-- Key schema fields
event_name          STRING
event_date          STRING        -- partition key (YYYYMMDD)
event_params        ARRAY<STRUCT<
  key               STRING,
  value             STRUCT<
    string_value    STRING,
    int_value       INT64,
    float_value     FLOAT64,
    double_value    FLOAT64
  >
>>
items               ARRAY<STRUCT<item_id, item_name, item_category,
                    quantity, item_revenue_in_usd, ...>>
ecommerce.purchase_revenue_in_usd  FLOAT64
user_pseudo_id      STRING
privacy_info.analytics_storage     STRING  -- Consent Mode

Extracting event_params (UNNEST pattern)

-- Extract a single parameter value
SELECT
  event_name,
  (SELECT value.string_value FROM UNNEST(event_params)
   WHERE key = 'clean_event')   AS clean_event,
  (SELECT value.int_value FROM UNNEST(event_params)
   WHERE key = 'ga_session_id') AS session_id
FROM `analytics_XXXXXXXXX.events_*`
WHERE _TABLE_SUFFIX BETWEEN '20201101' AND '20231031'
LIMIT 1000;

BigQuery Cost Comparison

ModeCostSLABest for
INTERACTIVE on-demand$6.25/TBImmediateAd-hoc analysis, dashboards
BATCH on-demand$6.25/TB6 hours maxSame cost — no saving on on-demand
BATCH flat-rate$0 (uses free shared slots)6 hours maxOvernight jobs, ML feature tables, exports
Query cache$0InstantRepeated identical queries within 24h
Common misconception: BATCH is not cheaper on on-demand pricing BATCH and INTERACTIVE scan the same bytes and cost the same $6.25/TB on on-demand. BATCH only saves money on flat-rate pricing, where it uses free shared community slots instead of consuming your reserved slot capacity. Use BATCH for overnight ML feature extraction jobs on flat-rate; use INTERACTIVE for live dashboards.

Cost Optimisation Rules

1
Always use _TABLE_SUFFIX bounds

Saves ~60% bytes vs full wildcard scan. Every query in this project uses BETWEEN '20201101' AND '20231031'.

2
Set maximum_bytes_billed

Add OPTIONS(maximum_bytes_billed=10737418240) — 10 GB safety cap — to every query in production.

3
Cluster ML feature tables by user_pseudo_id

Reduces bytes scanned by ~30% on all user-level join queries. Set at CREATE TABLE time.

Feature Engineering (BigQuery → Python)

-- Feature extraction query (Query 10 in bigquery_queries.sql)
SELECT
  user_pseudo_id,
  COUNT(DISTINCT ...) AS session_count,
  ANY_VALUE(device.category)                  AS device_category,
  ANY_VALUE(traffic_source.medium)            AS traffic_medium,
  MAX(...engagement_time_msec)                AS max_engagement_ms,
  COUNTIF(event_name = 'view_item')          AS view_item_count,
  COUNTIF(event_name = 'add_to_cart')        AS add_to_cart_count,
  MAX(IF(event_name='purchase',1,0))          AS did_purchase  -- label
FROM `analytics_XXXXXXXXX.events_*`
WHERE _TABLE_SUFFIX BETWEEN '20201101' AND '20231031'
GROUP BY user_pseudo_id;

GA4 Best Practices

1
Use gtag.js consent defaults before GA4 Config fires

Consent Mode v2 requires 4 signals: analytics_storage, ad_storage, ad_user_data, ad_personalization. All default to 'denied'. CMP updates them after user consent. Use wait_for_update: 500ms.

2
Validate with DebugView before publishing

GA4 DebugView shows events in real time. Set debug_mode=1 via GTM Preview Mode to activate. Cross-reference with BigQuery — confirmed debug_mode=1 in Nov 2020 data from this dataset.

3
Set data retention to 14 months, use BigQuery for longer history

Free GA4 caps at 14 months. The full 3-year dataset in this project lives in BigQuery. Never rely on the GA4 UI for historical trend analysis beyond 14 months.

GTM Best Practices

1
Store all GTM containers in Git

Export container JSON after every publish. Version control lets you roll back a bad tag deploy in seconds. Many agencies use CI/CD pipelines to deploy GTM containers automatically.

2
Always clear ecommerce: null before each dataLayer push

Prevents data bleeding between events. This is the single most common GTM ecommerce implementation bug.
